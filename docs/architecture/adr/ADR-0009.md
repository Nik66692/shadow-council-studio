# ADR-0009 — Local-first SQLite with Supabase as optional remote authority

- Status: Accepted for Phase 1.6 foundation
- Date: 2026-07-21
- Decision owner: Niccolò

## Context

Shadow Council Studio already uses a local SQLite database for canonical import evidence, human review, audit and Database Studio. The project now needs a remotely accessible database that can later support a web client, authenticated collaborators, backups and shared project data.

Replacing SQLite outright would make the desktop app depend on network availability and would jeopardize the current offline workflow. Treating both databases as unrestricted peers would create ambiguous authority and unsafe conflict resolution.

## Decision

The system adopts a local-first architecture:

- SQLite remains the desktop application's local database, offline cache and durable mutation queue.
- Supabase PostgreSQL becomes the optional official remote workspace datastore after a project is configured and the user authenticates.
- Supabase Auth identifies users. PostgreSQL Row Level Security authorizes access per workspace membership.
- Only a Supabase publishable key may be stored or shipped in the desktop app. Secret and service-role keys are forbidden.
- Every remotely exposed project table contains `workspace_id` and is protected by RLS.
- Cloud writes are explicit in Phase 1.6. Automatic two-way synchronization is not enabled until conflict rules and entity-specific merge semantics are approved.
- Immutable evidence and decision records remain append-only remotely.
- Local source text, source hashes and approval provenance are never rewritten by synchronization.

## Authority model

1. The immutable Source of Truth file and manifest remain the documentary authority for imported canon evidence.
2. Human-approved canon entries are authoritative application records.
3. Supabase is the official shared copy once cloud mode is enabled.
4. SQLite remains a valid offline working copy and retains unsynchronized mutations in an outbox.
5. A conflict is never resolved silently. Conflicting local and remote versions are recorded for explicit resolution.

## Security model

- Desktop clients use a publishable key plus the authenticated user's JWT.
- The secret/service-role key is never present in source control, application resources, SQLite or frontend state.
- RLS policies require workspace membership for reads and owner/editor role for writes.
- Workspace creation is performed through a narrow `security definer` function that atomically creates the workspace and its first owner membership.
- Audit and review decision tables are append-only through the client-facing API.

## Synchronization phases

### Phase 1.6

- Store optional project URL, publishable key and selected workspace locally.
- Authenticate and list/create accessible workspaces.
- Add local outbox, cursor and conflict tables.
- Add remote schema, RLS and sync metadata.
- Do not upload existing local data automatically.

### Later phase

- Register devices.
- Push idempotent outbox mutations using `mutation_key`.
- Pull remote changes by ordered cursor.
- Detect row-version conflicts.
- Apply entity-specific conflict rules approved in a separate ADR.

## Consequences

### Positive

- Existing offline behavior is preserved.
- A future web app can share the same authenticated PostgreSQL backend.
- Supabase Studio can be used for technical schema and data inspection.
- RLS prevents one workspace from reading another workspace's records.
- Migration SQL is versioned in Git and can be reproduced locally with the Supabase CLI.

### Costs and risks

- Two datastores require explicit synchronization design.
- Authentication and token lifecycle become part of the desktop application.
- Conflict resolution cannot be generic for all Shadow Council entities.
- A live project still has to be provisioned and its migration deployed outside the repository.

## Rejected alternatives

- Supabase-only desktop app: rejected because it removes offline operation.
- SQLite file stored in cloud storage: rejected because concurrent access and relational integrity would be unsafe.
- Service-role key embedded in Tauri: rejected because it bypasses RLS and would compromise the entire project.
- Automatic last-write-wins: rejected because it could silently overwrite canon decisions and evidence.
